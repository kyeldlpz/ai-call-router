# Code Quality Skills — RecoverAi

## Purpose

Enforce SonarQube-inspired clean code standards that prevent technical debt accumulation even under hackathon speed. These rules apply to all code generated by humans and AI agents.

---

## No Duplicate Code

### Rules

1. If you write the same logic twice, extract it into a shared function
2. If you write the same UI pattern twice, extract it into a component
3. If you write the same API call pattern twice, extract it into a client utility
4. Exception: test code may repeat setup patterns for readability
5. "Same logic" means identical intent, not just identical characters

### Example (Violation)

```typescript
// BAD: Same timestamp formatting in two components
// In TranscriptMessage:
const time = new Date(timestamp).toLocaleTimeString("en-US", { hour: "2-digit", minute: "2-digit", second: "2-digit" });

// In CallDuration:
const time = new Date(startedAt).toLocaleTimeString("en-US", { hour: "2-digit", minute: "2-digit", second: "2-digit" });
```

### Example (Correct)

```typescript
// lib/utils.ts
export function formatTime(isoString: string): string {
  return new Date(isoString).toLocaleTimeString("en-US", {
    hour: "2-digit",
    minute: "2-digit",
    second: "2-digit",
  });
}
```

### Detection

Before writing a function, search the codebase for similar logic. If it exists, reuse or refactor.

## No Magic Numbers

### Rules

1. Every numeric literal must have a named constant explaining its purpose
2. Every string literal used as a configuration value must be a named constant
3. Exceptions: 0, 1, -1 in obvious contexts (array indexing, boolean checks)
4. Constants defined at module level or in a dedicated constants file

### Example (Violation)

```python
# BAD: What does 500 mean? What does 0.5 mean?
await asyncio.sleep(0.5)
if len(transcript) > 500:
    truncate(transcript)
```

### Example (Correct)

```python
# Good: Self-documenting
RETRY_DELAY_SECONDS = 0.5
MAX_TRANSCRIPT_TOKENS = 500

await asyncio.sleep(RETRY_DELAY_SECONDS)
if len(transcript) > MAX_TRANSCRIPT_TOKENS:
    truncate(transcript)
```

```typescript
// Good: Named constants
const AUTO_SCROLL_DELAY_MS = 100;
const MAX_RECONNECT_ATTEMPTS = 3;
const WS_HEARTBEAT_INTERVAL_MS = 30_000;
```

## No Nested Complexity

### Rules

1. Maximum 3 levels of nesting (if/for/try)
2. Use early returns to reduce nesting
3. Use guard clauses at the top of functions
4. Extract nested logic into named helper functions
5. Avoid nested ternaries in JSX — use variables or early returns

### Example (Violation)

```python
# BAD: 4 levels of nesting
async def handle_event(event):
    if event:
        if event.get("type"):
            if event["type"] == "response.audio_transcript.delta":
                if event.get("delta"):
                    process_delta(event["delta"])
```

### Example (Correct)

```python
# Good: Guard clauses + early return
async def handle_event(event: dict) -> None:
    if not event or not event.get("type"):
        return

    event_type = event["type"]
    if event_type != "response.audio_transcript.delta":
        return

    delta = event.get("delta")
    if delta:
        process_delta(delta)
```

### JSX Nesting Rule

```tsx
// BAD: Nested ternary
{status === "idle" ? <StartButton /> : status === "active" ? <EndButton /> : status === "error" ? <RetryButton /> : null}

// GOOD: Extract to variable or function
function CallActionButton({ status }: { status: CallStatus }) {
  if (status === "idle") return <StartButton />;
  if (status === "active") return <EndButton />;
  if (status === "error") return <RetryButton />;
  return null;
}
```

## No Large Functions

### Rules

1. Maximum 20 lines of logic per function (excluding type annotations and blank lines)
2. If a function scrolls past one screen, it's too long
3. Each function does one thing — describable in a single sentence without "and"
4. Complex functions are composed from smaller named functions

### Example (Violation)

```typescript
// BAD: 40+ lines doing multiple things
async function startCall() {
  // Create call via API (lines 1-8)
  // Connect WebSocket (lines 9-18)
  // Start audio capture (lines 19-28)
  // Set up event handlers (lines 29-45)
}
```

### Example (Correct)

```typescript
// Good: Composed from focused functions
async function startCall() {
  const callId = await createCallSession();
  const ws = await connectWebSocket(callId);
  const audio = await startAudioCapture();
  setupEventHandlers(ws, audio);
  dispatch({ type: "CALL_CONNECTED" });
}
```

## Naming Conventions

### Rules

1. Names describe WHAT, not HOW
2. Boolean variables start with `is`, `has`, `can`, `should`
3. Functions start with a verb: `create`, `get`, `parse`, `handle`, `format`
4. Event handlers start with `handle` or `on`: `handleCallStart`, `onMessage`
5. No abbreviations unless universally understood: `ws` (WebSocket), `ctx` (context), `msg` (message)
6. No single-letter variables except loop indices (`i`, `j`) and lambda params (`x => x.id`)

### Naming Quality Check

| Bad Name | Why | Good Name |
|----------|-----|-----------|
| `data` | Too generic | `callResponse` |
| `temp` | Meaningless | `parsedEvent` |
| `flag` | What flag? | `isConnected` |
| `process` | Process what? | `parseTranscriptEvent` |
| `utils2` | Numbered files | `websocket-utils` |
| `handleStuff` | Vague | `handleWebSocketMessage` |
| `x` | Anonymous | `transcriptEntry` |

### File Naming

```
# Frontend (kebab-case)
call-panel.tsx          ✓
CallPanel.tsx           ✗
callPanel.tsx           ✗

# Backend (snake_case)
voice_intake.py         ✓
voiceIntake.py          ✗
VoiceIntake.py          ✗
```

## Maintainability Standards

### Rules

1. Every file has a clear single responsibility
2. Every module's public API is obvious (exports are intentional)
3. Circular imports are forbidden — restructure if detected
4. Dead code is deleted immediately — git preserves history
5. Comments explain WHY, not WHAT (the code explains what)
6. Type annotations on all public function signatures

### Comment Standards

```python
# BAD: Comments that describe what code does
# Increment counter by 1
counter += 1

# GOOD: Comments that explain why
# OpenAI Realtime API requires a 300ms buffer before sending audio
# to avoid clipping the start of speech
await asyncio.sleep(0.3)
```

```typescript
// BAD: Obvious comment
// Check if call is active
if (status === "active") { ... }

// GOOD: Explains non-obvious decision
// Auto-scroll disabled during manual scrollback to avoid jarring UX
if (!userHasScrolled) {
  scrollToBottom();
}
```

## Readability Standards

### Rules

1. Line length: max 100 characters (Python), max 120 characters (TypeScript)
2. One statement per line — no chained logic on single lines
3. Blank lines separate logical groups within a function
4. Import statements grouped: external → internal → types
5. Consistent quote style: double quotes in TypeScript, double quotes in Python
6. Trailing commas in multi-line structures (enables cleaner diffs)

### Import Organization

```typescript
// 1. External libraries
import { useState, useCallback } from "react";
import { Card } from "@/components/ui/card";

// 2. Internal modules
import { useCallContext } from "@/context/call-context";
import { formatTime } from "@/lib/utils";

// 3. Types (type-only imports)
import type { CallStatus, TranscriptMessage } from "@/types";
```

```python
# 1. Standard library
import json
import asyncio
from datetime import datetime

# 2. Third-party
from fastapi import WebSocket
from pydantic import BaseModel

# 3. Internal
from app.config import settings
from app.models.call import CallSession
```

## Refactoring Requirements

### When to Refactor

1. Before adding a feature to messy code — clean first, then add
2. When a function exceeds 20 lines
3. When a file exceeds its size limit
4. When you spot duplicate logic during development
5. When a name no longer describes what the code does

### When NOT to Refactor

1. Do not refactor working code for aesthetic preference during hackathon
2. Do not refactor code you don't understand without testing
3. Do not refactor across module boundaries unless you own both modules
4. Do not refactor in the same commit as a feature addition

### Refactoring Checklist

- [ ] Does the refactored code pass the same tests?
- [ ] Does the demo still work end-to-end?
- [ ] Are all imports updated?
- [ ] Are there no dead code remnants?
- [ ] Is the commit message clear about what was refactored and why?
